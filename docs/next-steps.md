# 下一阶段路线

当前版本是前端流程原型：页面结构、核心入口、主要交互已经成型，数据仍然是本地 mock。下一阶段建议按“演示可用 -> 真实能力 -> 产品化”的顺序推进。

## 1. 演示可用版本

目标：让投资人、合作方或内部团队可以打开链接，快速理解家长端体验。

- 完善首屏关键状态：孩子在线状态、电量、定位、今日任务、快捷呼叫。
- 把弹窗里的占位能力改成更完整的演示流程，例如添加守护区域、查看轨迹详情、生成周报建议。
- 补齐空状态、加载状态和失败提示，避免演示时出现“点了没反应”的感觉。
- 增加少量真实感数据场景：上学日、放学路上、睡前管控、周末娱乐时间。
- 做一次移动端适配检查，重点看小屏手机上的按钮、弹窗和底部导航。

## 2. 真实能力接入

目标：从纯原型转成可和后端、设备、AI 服务对接的应用骨架。

- 账号与家庭关系：家长登录、孩子档案、家庭成员权限。
- 设备绑定：扫码/绑定码、设备在线状态、电量、号码、SIM 流量。
- 位置安全：实时定位、轨迹列表、电子围栏、到离校提醒。
- 通讯管控：白名单联系人、陌生人拒接、通话记录、紧急呼叫。
- 任务激励：任务创建、提醒同步、完成确认、星能量规则。
- AI Agent：自然语言创建提醒、总结孩子状态、生成周报、解释异常提醒。

## 3. 产品化与合规

目标：让产品进入可试用、可运营、可持续迭代的状态。

- 儿童隐私与家长授权：明确数据用途、留存周期、删除入口。
- 安全边界：定位和通话能力需要显式授权、操作留痕和异常提醒。
- 运营配置：任务模板、成长报告模板、设备型号能力差异。
- 可观测性：记录关键操作成功率、页面访问、接口失败原因。
- 版本策略：区分演示环境、测试环境和正式环境，避免 mock 数据混入正式包。

## 建议优先级

1. 先把 GitHub Pages 链接跑通，形成随时可分享的在线原型。
2. 再把“工作台 + 安全”做成第一条完整演示链路。
3. 然后接入最小后端：账号、孩子档案、设备状态、定位 mock API。
4. 最后再扩展 AI Agent 和成长报告，避免过早把复杂能力堆进前端。
